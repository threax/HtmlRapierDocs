Data Attributes

At its heart this library is a fancy way of labeling elements on a html page with some attributes that make extracting them later, with some special properties, easier. You can define the following types of attributes on your elements.

  • Controller - data-hr-controller="controllerName" - This defines the primary interface between your javascript and the markup and act as more or less of an entry point for everything else. The controllerName value defines the name of the controller.
  • Event - data-hr-on-eventName="functionName" - Bind the html event eventName to the function in your controller named functionName. This supports all events that the browser can register with addEventListener.
  • Config - data-hr-config-name="value" - Create a configuration for the controller named name with the value value. All the config values for a controller are extracted into a single object so the -name part should be unique.
  • Toggle - data-hr-toggle="name" - Create a toggle named name. Toggles abstract the dom from your javascript for things like styles and classes. Toggles can be set to on and off in your javascript code. You can use the following attributes to in addition to a data-hr-toggle on an element to control what it looks like when the toggle is on and off.
    • On Styles - data-hr-style-on="styleList" - Apply the styles in styleList when this toggle is on.
    • Off Styles - data-hr-style-off="styleList" - Apply the styles in styleList when this toggle is off.
    • On Classes - data-hr-class-on="classList" - Apply the classes in classList when this toggle is on.
    • Off Classes - data-hr-class-off="classList" - Apply the classes in classList when this toggle is off.
    • Idle Classes - data-hr-class-idle="classList" - Apply classList classes when the animations on this element have completed. HtmlRapier can track animation states when you toggle and will apply this class after the animations are completed.
    • You can also make your own modes for styles or classes. The mode name is derived from the last section of the name. For example you could have an element with data-hr-style-disabled added to -on and -off styles. This will create a third toggle mode for "disabled". Note that for additional modes to be supported the Toggle defined in the typescript code be aware of the mode. You can make custom toggle classes to support this.
  • Form - data-hr-form="name" - Create a form on the specified element named name. Forms allow you to easily read / write data from form elements in the dom. In addition to this you can set a prototype object that will be used as the prototype object for data returned from the form. This allows you to create forms with extra data not on the form easily since it can be stored in the prototype object. You can also set a schema for the form. Currently this is loosly based on JsonSchema4, but there are extensions to the data to help fully describe the forms. You can let the schema build the form or you can provide the elements yourself and fill in dynamic things like combo boxes, or you can define some elements and have others created automaticly from the schema.
    • If you are defining a form element yourself you can create a toggle named itemNameError (where itemName is the name set on the form attribute) to be able to toggle things with an error. You can also define a view itemNameErrorMessage that has an argument of {{this}} somewhere in it. The error messages are passed directly so you want to use {{this}}. All of these elements should be inside an element that has a data-hr-input-start attribute on it. You can also dictate where the auto generated elements should insert by specifying a data-hr-form-end attribute, otherwise they will insert at the end of what is on the form already.
    • Custom Json Schema Extensions
      • x-ui-order - This determines the ui order of the properties, the properties will be sorted by this index before they are processed.
      • x-ui-type - Specify the type of control that should be created, this is not a 1:1 mapping to html input types, but it is close. You can create special types and create template variants for types you want more control over the appearance of on your form.
      • x-values - An array of label value pairs that specify possible values for the control.
      • x-enumNames - If an enum string is provided, this is an array of display names for those enum values.
      • x-value - If there is a single value for a field this can set that value. Helpful for things like checkboxes.
  • View - data-hr-view="name" - Create a view on the specified element named name. A view, if it is able, will automatically repeat collection data that it is passed, otherwise it outputs a single element. All views will escape their output for malicious html, so you don't need to scrub your data yourself. Depending on what kind of element you attach this to you will get different concrete view types, however, the interface remains the same and you could swap one for another without changing code.
    • ComponentModel - This model is created when the data-hr-model attribute is added to a node that also has data-hr-component on it (or will get one from an anonymous template). It will use a template that you define to output the data you specify. If the data is a collection then you will get repeated elements. This is the most flexible kind of model. To get this kind of model add a data-hr-view-component="name" attribute to it or define a template as its child, which will make an anonymous template, which has the same effect without you needing to come up with a name yourself.
    • TextNodeModel - With no special conditions a TextNodeModel is created. This will look for variables defined like {{varName}} and fill them in with matching data on the object bound to this model.
  • Model - data-hr-model="name" - This is the old name for forms and views and is deprecated.  Originally these were handled as one thing, but it didn't really make sense. For now the system still supports getting models from binding collecitons and using data-hr-model in html, however, those things are deprecated and should never be used in new code.
  • Component - <template data-hr-component="name">markup</template> - Create a component named name with the html code markup. Components must be <template> elements. You can actually omit the data-hr-component="name" attribute if you are creating the component as a direct descendant of a model node, which creates an anonymous template that gets its name assigned at runtime. This makes labeling components to their models much simpler since you don't have to come up with a ton of unique names, however, sometimes you will need to specify the names directly and you can do this here.
    • Variant - <template data-hr-component="componentName" data-hr-variant="name">markup</template> - This will specify a variant of a template. Variants allow you to have different verisons of a component depending on your source data. You will define a function when binding data to a model that returns the name of the variant to use for a component. Variant names must be unique for each component, but can be repeated otherwise. You can omit the data-hr-component part if you are creating this variant as part of a variant chain. To do this create your first template element, either anonymous or with a name. Next as the immediate sibling to that first <template> element create another with the data-hr-variant attribute and give it a variant name. This will attach the 2nd template as a variant on the first. This chain will continue until there are no more siblings or another element type is encountered.
  • Handle - data-hr-handle - Get direct access to the HtmlElement pointed to by the named handle. Mostly useful to pass elements into third party libraries that want them, otherwise should be avoided as much as possible.
  • Ignore - data-hr-ignored - Add this to an element to ignore any components and controllers it has as children. This is mostly a special case to handle editors where you still need to run other HtmlRapier code.
  • Run - data-hr-run - This is used to make jsns run a particular named javascript module. If you are using jsns (the default) you will need one of these in your code to kick off whatever runs your main code, otherwise you can do whatever is appropriate for your package manager. HtmlRapier does not mandate a project layout (beyond using typescript ideally in a flat dependency structure, and even then you can make it work in pure javascript).
